# mempool 性能优化对比说明（`old_project` → `project`）

> 对比范围：`F:/projects/memorypool/old_project`（优化前） vs `F:/projects/memorypool/project`（优化后）
> 核心目标：解释“做了哪些优化、为什么会更快、代价是什么、如何验证”

---

## 1. 总览：优化带来的结果

从你现有统计文档可见，优化后版本在吞吐和扩展性上有明显提升：

- 单线程：在 16B~4096B 各 size class 上，`project` 相比 `old_project` 从“中大块明显落后系统 malloc”变成“全尺寸领先系统 malloc”。
- 多线程：8 线程吞吐从约 **22,363 ops/ms** 提升到约 **93,536 ops/ms**，扩展性从接近不增长提升到显著增长。
- 核心原因不是单点微调，而是 **页面管理策略 + TLC 补货策略 + bitmap 扫描策略** 三个路径同时优化。

---

## 2. 代码层面的关键改动（逐项对比）

## 2.1 Arena 位图扫描从“朴素全扫”升级为“hint + 快速跳词”

### 优化前
- `old_project/src/arena.cpp` 的 `find_free_pages` 从 0 开始线性扫描整个 bitmap。
- 每次分配都可能重复扫已经满的区域，chunk 越满越容易重复无效扫描。

### 优化后
- `project/src/arena.h` 给 `ChunkHeader` 增加 `next_free_hint` 字段。
- `project/src/arena.cpp` 的 `find_free_pages`：
  - 从 `next_free_hint` 开始扫描；
  - 对“整词全满（`~0ULL`）”场景直接跳过 64 bit；
  - 扫到末尾后再 wrap 到 `[0, hint)` 补扫，保证正确性。
- 分配成功后更新 `next_free_hint` 到已分配区后面；释放时若释放位置更小则回退 hint。

### 性能收益机制
- 降低重复扫描成本，减少在高占用 chunk 中的无效检查。
- 让分配热点更接近“局部前进”，避免每次从头扫描。

---

## 2.2 页面生命周期策略改变：从“free 时 decommit”改为“保留 committed”

### 优化前
- `old_project/src/arena.cpp` 在 `arena_free_pages` 中调用 `platform::mem_decommit`。
- 后续重用该页时，在 `arena_alloc_pages` 中必须 `platform::mem_recommit`。

### 优化后
- `project/src/arena.cpp` 在 `arena_free_pages` 中**不再 decommit**（代码注释已明确说明这是有意策略）。
- 因为页保持 committed，重用时不再需要 recommit 才可访问。

### 性能收益机制
- 去掉大量 `VirtualFree(MEM_DECOMMIT)` / `VirtualAlloc(MEM_COMMIT)` 往返系统调用。
- 对频繁分配释放的工作负载（尤其中大块）提升显著。

### 代价
- 峰值物理内存回收会更慢（页会在 chunk 生命周期内保持可用状态）。
- 这是典型“以空间换时间”的工程取舍。

---

## 2.3 TLC 补货从“单页补货”升级为“按目标块数批量补货”

### 优化前
- `old_project/src/tlc.h` 的 bucket 只记录 `page_ptr + page_count`（通常 1 页）。
- `old_project/src/tlc.cpp` 的 `bucket_alloc_new_page` 每次仅向 arena 申请 1 页。

### 优化后
- `project/src/tlc.h` 将 bucket 的页跟踪改为 `PageRange* page_list`，支持多批次、多页段。
- `project/src/tlc.cpp` 增加：
  - `calc_batch_pages`：按 `MP_REFILL_BLOCKS` 计算单次应补多少页（上限 64 页）；
  - `init_multi_page_freelist`：一次初始化多页 free list；
  - `bucket_alloc_new_pages`：优先批量申请，失败再退化到 1 页。

### 性能收益机制
- 大幅减少 slow path 次数和 arena 锁进入次数。
- 对大 size class（每页 block 少）收益尤其明显。
- 分配路径更快进入“稳态快速路径（free_head pop）”。

### 代价
- 短时缓存块增多，可能提高线程局部暂存内存。
- `tlc_flush` 需要遍历 `page_list` 逐段归还页面，逻辑复杂度略增。

---

## 2.4 TLC 回收路径适配批量策略

### 优化前
- `tlc_flush` 只归还 bucket 上单个 `page_ptr/page_count`。

### 优化后
- `tlc_flush` 遍历 `PageRange` 链表，逐段调用 `arena_free_pages`。
- `tlc_destroy` 额外兜底释放 page range 跟踪节点。

### 影响
- 与批量补货配套，确保线程退出或 detach 时不会泄漏归还信息。

---

## 2.5 跨线程 free 路径本身基本保持不变，但整体受益于上游优化

`mp_free` 的同线程/跨线程分流（owner thread 判断 + remote CAS push）在两版中基本一致。
但因为“批量补货 + 页面不 decommit”降低了慢路径频率，跨线程场景下最终也会间接受益。

---

## 3. 哪些模块几乎没变

- `mempool.cpp` 主体 API 流程（init/shutdown/malloc/free/realloc/arena API）两版基本一致。
- `platform_win32.cpp`、`platform_linux.cpp` 的基础平台抽象接口未发生决定性性能路径改动（仍保留 decommit/recommit 能力，但优化后核心路径减少了对它们的调用）。
- `internal.h` 参数（如 `MP_REFILL_BLOCKS=64`, `MP_HIGH_WATERMARK=256`）两版一致，重点是“参数真正被怎样利用”的策略变化。

---

## 4. 性能数据变化与原因映射

结合你两份 `docs/项目统计信息.md`，可以建立以下映射：

1. **中大块（256B/1024B/4096B）提升最显著**
   主要来自：批量补货减少 arena 慢路径 + 去掉 decommit/recommit 往返。

2. **多线程扩展性大幅提升**
   主要来自：每线程 TLC 单次补货量增加，减少共享 arena 锁争用频次。

3. **free 路径持续很快**
   两版都走 local_free/thread_free 设计，但优化后 alloc 端不再频繁被系统页操作拖慢，整体 alloc+free 周期更平衡。

---

## 5. 优化前后架构差异（便于答辩展示）

### 优化前（old_project）
- 分配慢路径：
  - arena bitmap 从头扫；
  - bucket 常常单页补货；
  - 页面回收时 decommit，重用时 recommit。
- 特点：实现直观，但系统调用和慢路径比例偏高。

### 优化后（project）
- 分配慢路径：
  - bitmap 从 hint 起扫，并能整词跳过；
  - bucket 以目标块数批量补货，失败再降级；
  - free 页保持 committed，减少页状态切换。
- 特点：牺牲部分“即时物理内存回收”，换取更高吞吐和更好并发扩展。

---

## 6. 工程取舍与风险点

1. **内存占用取舍**
   不 decommit 会让进程在高峰后更“粘住”一部分物理页，但能稳定降低分配抖动。

2. **实现复杂度上升**
   `PageRange`、批量补货、hint 扫描都提高了维护成本，需要更强测试覆盖。

3. **参数敏感性**
   `MP_REFILL_BLOCKS=64`、批量上限 64 页是经验值。不同 workload 可能需要调参。

## 7. 建议的验证方法（你可直接写到论文实验章节）

1. **吞吐基准（已完成）**
   - 保持当前 `test_perf`，分别测 old/new 在 16B~4096B 上的 alloc/free/总耗时。

2. **慢路径频率对比**
   - 统计 `slow_path_hits`、arena 页申请次数，验证批量补货是否显著减少慢路径。

3. **锁争用对比（多线程）**
   - 在 1/2/4/8 线程下对比 arena mutex 进入次数（可通过计数器或 profiler）。

4. **内存占用曲线**
   - 对比 old/new 在高峰后空闲阶段的 RSS/工作集，量化“不 decommit”带来的空间代价。

---

## 8. 一句话总结（答辩可用）

本次性能优化的本质是：**把频繁、小粒度、昂贵的慢路径操作（全量扫描、单页补货、decommit/recommit）改造成更“批处理”和更“局部性友好”的策略（hint 扫描、批量补货、保留 committed 页）**，从而在不改动外部 API 的前提下显著提升了吞吐和并发扩展能力。

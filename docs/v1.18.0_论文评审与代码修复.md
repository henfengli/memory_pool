## 代码修复

### 1. init_tls_slot 加 std::call_once 保护（真问题）

**问题**: `src/tlc.cpp` 中 `init_tls_slot` 直接判断 `g_tlc_tls_index == TLS_OUT_OF_INDEXES` 后调用 `TlsAlloc()`，没有同步保护。两个线程同时首次进入时可能都看到 `TLS_OUT_OF_INDEXES`，都调用 `TlsAlloc()`，导致 TLS 槽泄漏。实际不出问题是因为主线程通常先于子线程调用 `mp_malloc`，但这是脆弱依赖。

**修复**: 加入 `std::once_flag g_tlc_tls_once`，用 `std::call_once` 保护初始化。

**文件**: `src/tlc.cpp` init_tls_slot 函数（17-26 行）

### 2. arena.cpp:268 显式 relaxed load（代码风格 cleanup）

**问题**: `find_free_pages` 的 wrap-around 分支使用 `chunk->page_bitmap[word]`（隐式 atomic load 默认 seq_cst），与同函数其他三处 `.load(std::memory_order_relaxed)` 不一致。x86 上无性能影响，ARM 上会插入 dmb 屏障。

**修复**: 改为 `chunk->page_bitmap[word].load(std::memory_order_relaxed)`，与其他位置对齐。

**文件**: `src/arena.cpp:268`

## 评审者错判说明

第四轮评审者声称的"严重问题"（bitmap 不是 atomic、arena_alloc_pages 全程加 mutex、Phase 1/2 是虚构）在第五轮中被评审者本人公开收回。代码 v1.16.0 起 page_bitmap 和 next_free_hint 均为 std::atomic，arena_alloc_pages 实现两阶段 CAS（Phase 1 无锁 + Phase 2 仅分配新 Chunk 时加 mutex）。论文 4.3.2 节描述与代码完全一致。

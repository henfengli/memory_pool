# v1.19.0 SizeClassInfo 编译期合并 + PageMeta 瘦身

日期: 2026-04-30
触发: 与用户讨论 `calc_batch_pages` 的工程合理性时，发现 size class 派生属性（block_size、blocks_per_page、batch_pages）散落在三处运行时计算（除法、分支、上限钳位），与 tcmalloc `SizeMap` / mimalloc `_mi_bin` 把全部派生属性预算到一张编译期表的工业做法不一致。

## 代码重构

### 1. 引入 `SizeClassInfo` 编译期表（src/size_class.h）

新增结构体：

```cpp
struct SizeClassInfo {
    uint16_t block_size;       // 16 ~ 4096
    uint16_t blocks_per_page;  // MP_PAGE_SIZE / block_size
    uint8_t  batch_pages;      // 慢路径一次申请几页
    uint8_t  _pad;
};
```

通过 `constexpr SCInfoTable` 在编译期把 25 档全部预算好。`compute_batch_pages` 由原 `tlc.cpp::calc_batch_pages` 提升为 `constexpr` 函数。慢路径 `bucket_alloc_new_pages` 现在只需一次 `sc_info(idx).batch_pages` 表查询，省下原先连续两次函数调用 + 一次除法。

所有调用点（mempool.cpp、tlc.cpp、arena.cpp）统一改用 `sc_info(idx).block_size` / `sc_info(idx).blocks_per_page` / `sc_info(idx).batch_pages` 直接访问结构体字段，不再保留 `sc_block_size` / `sc_blocks_per_page` / `sc_batch_pages` 这三个 thin wrapper。删除冗余间接层后，每个调用点都一目了然地表达"从 sc_info 表里读哪一个字段"。

### 2. 删除 `PageMeta::block_count` 字段（src/arena.h）

`block_count` 始终等于 `sc_info(bucket_idx).blocks_per_page`，且仅在 `bucket_try_reclaim_pages` 的回收判定中读一次（每 64 次 collect 才触发的冷路径）。

收益：
- PageMeta 由 6 字段（24B）减为 5 字段 + 2B 对齐填充（仍 24B 但消除一个冗余写）。
- arena.cpp 4 处 `pm.block_count = (uint16_t)blk_per_page;` 初始化全部删除。
- arena_alloc_pages 中 `uint32_t blk_per_page = sc_info(bucket_idx).blocks_per_page;` 局部变量随之删除。
- 消除"初始化值与 sc_info 漂移"的潜在 bug：唯一真值源就是 `sc_info`。

`tlc.cpp:156` 的回收判定改为 `pm.freed_count < sc_info(pm.bucket_idx).blocks_per_page`。冷路径增加一次 O(1) 表查询（≤2 ns），在每 64 次 collect 才触发的回收扫描中完全可忽略。

### 3. 删除 `calc_batch_pages` 函数（src/tlc.cpp）

原 L128-133 整段删除。逻辑提升到 `size_class.h::detail::compute_batch_pages` 的 constexpr 上下文中，编译期就被 SCInfoTable 构造器消耗掉，不留运行时函数符号。

### 4. 顺带清理：删除 thin wrapper（src/size_class.h）

`sc_block_size` / `sc_blocks_per_page` / `sc_batch_pages` 这三个一行 wrapper 是首次重构时为"调用点零修改"留的便宜事——但它们既不抽象也不简化，调用点 `sc_block_size(idx)` 与 `sc_info(idx).block_size` 读起来一样长，多一层间接反而模糊了"派生属性都聚合在 sc_info 表"这一意图。**全部删除**，6 个调用点（mempool.cpp 2 处、tlc.cpp 3 处、arena.cpp 1 处）改为直接读 `sc_info(idx).<field>`。

### 5. 顺带清理：删除死字段链（PageMeta::owner_thread → TLC::thread_id → get_thread_id）

代码审视中发现一条**完全只写不读**的字段链：
- `PageMeta::owner_thread`（[arena.h:20](../src/arena.h#L20)）：5 处赋值（arena.cpp 4 处 init + 1 处清零），零处读取。
- `TLC::thread_id`（[tlc.h:43](../src/tlc.h#L43)）：1 处赋值（[tlc.cpp:95](../src/tlc.cpp#L95)），仅作为 `arena_alloc_pages` 入参传给 `PageMeta::owner_thread`。
- `arena_alloc_pages` 的 `uint64_t thread_id` 入参（[arena.h:90](../src/arena.h#L90)、[arena.cpp:373](../src/arena.cpp#L373)）：仅供写 owner_thread。
- `platform::get_thread_id()`（[platform.h:25](../src/platform.h#L25)、Win32/Linux 各一份实现）：仅在 tlc 初始化时被调一次。

**整条链都是死代码**——线程归属判定全靠 `PageMeta::owner_tlc` 指针比较（[mempool.cpp:171](../src/mempool.cpp#L171)），thread_id 整数比较从未发生过。本次一并删除整条链：
- PageMeta 字段：8B/页 × 1008 页/chunk = **8KB 元数据/chunk** 立即省下；
- arena_alloc_pages 函数签名缩短一个参数；
- TLC 结构少 8 字节；
- platform 层删除 `get_thread_id()` 接口及两份平台实现。

### 6. 顺带清理：抽出重复的 freed_count 补偿循环（src/tlc.cpp）

`bucket_collect_thread_free` 和 `tlc_flush` 各有一段完全相同的 7 行循环，遍历 thread_free 链表对每个 block 所属 page 的 freed_count 自增。抽为 `compensate_freed_counts(FreeBlock* head)` inline helper，两处调用收敛到同一段实现。这是真冗余——未来要改 freed_count 补偿语义时只需改一处。

## 设计权衡说明

- **`Bucket::block_size` 保留**：它在 fast path（[tlc.cpp:278,283](../src/tlc.cpp#L278)）的 bump 分配中被读，删除会在最热路径上多一次 `sc_info[idx]` 间接寻址。25 个 bucket × 2 字节 = 50 字节/TLC 的节省不值这个风险。tcmalloc `ThreadCache::FreeList` 同样缓存 `size_`，mimalloc `mi_page_t` 也缓存 `block_size`。
- **`PageMeta::block_size` 保留**：被 [arena.cpp:520](../src/arena.cpp#L520)、[mempool.cpp:149](../src/mempool.cpp#L149) 用作"该页是否已分配"的哨兵（0 = 空闲），且 mempool.cpp:160 debug fill 也读它。删除需要换一套哨兵机制（如 `bucket_idx == 0xFFFF`），改动面超出本次重构边界。
- **不抽 PageMeta 初始化函数**：arena.cpp 4 处 PageMeta 初始化虽然结构相同，但分别绑定在不同的并发分支（CAS 成功 / CAS retry / double-check / 新 chunk）紧邻独立 return 路径。删 owner_thread 后每段只剩 4 行字段赋值，抽函数引入的函数签名 + 调用代价反而大于 4 行赋值本身的可读性。

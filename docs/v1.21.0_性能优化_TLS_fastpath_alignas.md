# v1.21.0: TLS 改 `__thread` / fast path 头内联 / Bucket cache-line 对齐

日期：2026-05-03
作者：henfengli + Claude (perf 实测分析)

## 一、起点：Linux 上跑 docs/性能指标测试.md 同口径 bench 输得离谱

v1.20.0 在 Linux + glibc 2.39 上跑 `real_tests/` 套件（与 `docs/性能指标测试.md` Win UCRT 同口径）：

| 测试 | docs Win UCRT | v1.20.0 Linux | 状态 |
|---|---|---|---|
| Table 1 batch 9 size | 1.20-1.57× | 1.29-1.77× | OK，量级一致 |
| **Table 2 K=256 churn 16-1024B** | **1.00-1.17×** | **0.32-0.40×** | **输 3×** |
| Table 3 多线程 64B T=8 | 1.18× | ~1.0× | 边缘 |

最关键的 alloc-free 稳态 churn 输系统 malloc 3 倍，这条线直接戳穿"mempool 比系统快"的核心论点。要命的是这条线 v1.20.0 之前一直没人测出来——Win UCRT 下因为 baseline 慢看不出来。

我之前的初版 docs/v1.20.0_实测重跑.md 把这归因于 "glibc 2.39 tcache 是世界级最强基线"——**这个结论是错的，没有 perf 数据支撑**。下面是怎么找出真正瓶颈的。

## 二、用 perf 找出真瓶颈

### 2.1 干掉 perf_event_paranoid 限制

VMware 默认 `kernel.perf_event_paranoid = 4`，连 cpu-clock 都不让普通用户用：

```bash
echo -1 | sudo tee /proc/sys/kernel/perf_event_paranoid
```

VMware 不暴露硬件 PMU（`<not supported>` for cycles/instructions），所以只能用软件事件 `cpu-clock`。**这导致没法看 cache miss、branch miss，但 hot function 分布还是能看的**。

### 2.2 写最小 perf 测试程序

`real_tests/perf_focused.cpp`：4M 次 K=256 ring churn 16B，单一 allocator 跑。

```bash
LD_LIBRARY_PATH=build_perf/bin \
  perf record -F 999 -e cpu-clock --call-graph fp \
  -o /tmp/perf_mp.data -- build_perf/bin/perf_focused mp
perf report -i /tmp/perf_mp.data --no-children --stdio --percent-limit 1
```

### 2.3 v1.20.0 mempool 的 hot 函数分布（修复前）

```
17.57%  libmempool.so  [.] mp::tlc_alloc
17.15%  libc.so.6      [.] pthread_getspecific@@GLIBC_2.34
15.06%  libc.so.6      [.] pthread_once@GLIBC_2.2.5
...
```

**32.21% 时间花在 pthread_getspecific + pthread_once**——这两个函数都是 libc 的 PLT 跳转。

对照同口径 glibc：

```
31.90%  libc.so.6      [.] _int_free
31.03%  libc.so.6      [.] malloc
22.41%  libc.so.6      [.] cfree
```

glibc 这边一次 pthread_* 都没有，时间全在 `__libc_malloc` / `_int_free` / `cfree` 自己的代码里。

### 2.4 核源码对比

mempool POSIX TLS（修复前 src/tlc.cpp:42-52）：

```cpp
static pthread_key_t g_tlc_key;
static pthread_once_t g_tlc_key_once = PTHREAD_ONCE_INIT;

TLC* tlc_current() {
    pthread_once(&g_tlc_key_once, make_tlc_key);
    return static_cast<TLC*>(pthread_getspecific(g_tlc_key));
}
```

每次 `mp_malloc` / `mp_free` 都进 `tlc_current()` → 两个 libc PLT 跳转。

glibc tcache（malloc/malloc.c:3124-3125）：

```c
static __thread bool tcache_shutting_down = false;
static __thread tcache_perthread_struct *tcache = NULL;
```

`__thread` 在 x86_64 上编译成单条 `mov %fs:offset, %reg`——零函数调用、零 PLT。

**这就是真瓶颈**。Win UCRT baseline 之所以输得不那么明显，是因为 Win 路径的 `tlc_current()` 只调一次 `TlsGetValue`，比 POSIX 的 `pthread_once + pthread_getspecific` 少一个 libc 调用。docs/Win UCRT 数据看不出问题，纯属 Windows 那边 mempool 自己代码就比 POSIX 路径快一半。

## 三、5 处修改（按贡献排序）

### Fix 1: POSIX TLS 改 `__thread` initial-exec（最大单一贡献）

**`src/tlc.cpp:42-57`**

```cpp
// 改前
static pthread_key_t g_tlc_key;
static pthread_once_t g_tlc_key_once = PTHREAD_ONCE_INIT;
TLC* tlc_current() {
    pthread_once(&g_tlc_key_once, make_tlc_key);
    return static_cast<TLC*>(pthread_getspecific(g_tlc_key));
}

// 改后
static __thread __attribute__((tls_model("initial-exec"))) TLC* g_current_tlc = nullptr;
TLC* tlc_current() { return g_current_tlc; }
```

**为什么 initial-exec**：默认 `__thread` 在 -fPIC 共享库里编译成 General Dynamic 模型，每次访问都调 `__tls_get_addr`，仍然是函数调用。`tls_model("initial-exec")` 强制用 Initial-Exec 模型，在 .so 加载时由动态链接器在静态 TLS 区分配 slot，访问时一条 `mov %fs:offset` 完事。代价是这个 .so **不能被 dlopen 加载**（必须随主程序启动加载），LD_PRELOAD 也不行。对一个允许的代价。

**实测影响**（K=256 churn 16B，4M iters）：
- 修前：mp 0.27s vs sys 0.07s（mempool 4× 慢）
- 修后：mp 0.13s vs sys 0.07s（mempool 略慢，下面还要继续修）

### Fix 2: fast path 头内联，slow path 留 .cpp

**`src/tlc.h` 新增**：

```cpp
MP_ALWAYS_INLINE inline void* tlc_alloc(TLC* tlc, uint32_t bucket_idx) {
    Bucket& b = tlc->buckets[bucket_idx];
    if (MP_LIKELY(b.free_head != nullptr)) {
        FreeBlock* blk = b.free_head;
        b.free_head = blk->next;
        if (blk->next) __builtin_prefetch(blk->next, 0, 3);
        return blk;
    }
    if (MP_LIKELY(b.local_free_head != nullptr)) {  // Fix 3 见下
        FreeBlock* blk = b.local_free_head;
        b.free_head = blk->next;
        b.local_free_head = nullptr;
        return blk;
    }
    return tlc_alloc_slow(tlc, bucket_idx);
}

MP_ALWAYS_INLINE inline void tlc_free(TLC* tlc, void* ptr, PageMeta*, uint32_t bucket_idx) {
    Bucket& b = tlc->buckets[bucket_idx];
    FreeBlock* blk = static_cast<FreeBlock*>(ptr);
    blk->next = b.local_free_head;
    b.local_free_head = blk;
}
```

`src/tlc.cpp` 把原 `tlc_alloc` 改名 `tlc_alloc_slow`，删掉头部的 free_head 分支（已挪到头）。

**为什么必须拆开**：原 `tlc_alloc` 函数体 1357 字节（`objdump` 看 symbol size = 0x55d），有 4 条路径（free_head pop / bump pointer / collect_local / collect_thread_free / alloc_new_pages），slow path 用了 6 个 callee-saved 寄存器（rbx, rbp, r12-r15）。

`perf annotate` 看到 fast path 也付了这 6 个 push/pop 的代价：

```
1fe0:   push   %rbp
1fe1:   mov    %rsp,%rbp
1fe4:   push   %r15
1fe6:   push   %r14
1fe8:   push   %r13
1fea:   mov    %esi,%r13d
1fed:   push   %r12
1fef:   imul   $0x38,%r13,%rcx   <- bucket index * sizeof(Bucket)
...
```

拆开后 fast path 在 mp_malloc 里直接展开，是 leaf 函数，无 push/pop。

### Fix 3: fast path 第二支 inline `local_free → free_head` swap

K=256 ring churn 的稳态：
- `mp_free` 一次：push 到 `b.local_free_head`
- `mp_malloc` 一次：从 `b.free_head` pop

但 alloc 和 free 用**两条不同的链表**！free_head 在初始 bump pointer 用完之后就一直空。

修前：每次 alloc 都进 `tlc_alloc_slow` → `bucket_collect_local` 把 local_free swap 到 free_head → 再递归调 `tlc_alloc`。perf 实测 36.63% 时间在 `tlc_alloc_slow`。

修后：把 `local_free → free_head` swap 直接 inline 到 `tlc_alloc` fast path 第二支（见 Fix 2 代码块）。整个 churn 路径：

```
mp_malloc:
  tlc = g_current_tlc           ; %fs:offset 一条 mov
  if (free_head) pop, return    ; 三条 mov
  if (local_free) swap, return  ; 三条 mov + 一次 store nullptr
mp_free:
  pm = chunk_of(ptr)            ; mask
  bucket_idx = pm->bucket_idx   ; load
  if (pm->owner_tlc == tlc):
    push to local_free          ; 两条 mov
```

`bucket_collect_local` 函数整个删掉，原本的 amortized reclaim 触发点（每 64 次 collect 调一次 reclaim）挪到 `bucket_alloc_new_pages`（拿新页时触发）。**对 churn 工作负载零 reclaim 影响**——churn 工作集完全驻留，从来不需要 reclaim 释放页。

### Fix 4: 冷路径 `__attribute__((noinline, cold))`

**`src/mempool.cpp:55`**

```cpp
__attribute__((noinline, cold))
static TLC* get_tlc() {
    ensure_init();
    TLC* tlc = tlc_current();
    if (MP_LIKELY(tlc != nullptr && tlc->arena != nullptr)) return tlc;
    return tlc_get_or_create(g_default_arena);
}
```

`mp_malloc` 中 `if (MP_UNLIKELY(!tlc || !tlc->arena)) tlc = get_tlc();` 这条冷路径会拉 `do_init` / `pthread_mutex_lock` / `tlc_get_or_create` 进 `mp_malloc` 的可达调用图，编译器只能为它建栈帧。把 `get_tlc` 标 noinline + cold 后，`mp_malloc` 编出来的 fast path 不再 `push %rbp; push %rbx`，直接走。

### Fix 5: `alignas(64)` Bucket + TLC

**`src/tlc.h`**

```cpp
struct alignas(64) Bucket { ... };
static_assert(sizeof(Bucket) == 64);

struct alignas(64) TLC { ... };
```

**两个收益**：

1. **`sizeof(Bucket)` 从 56 → 64 是 power-of-2**，索引 Bucket 的代码从 `imul $0x38, %rdx, %rdx` 变成 `shl $0x6, %rax`。imul 在 x86 上 3 cycle 延迟，shl 1 cycle。每次 alloc + free 一对省 4 cycle。
2. **TLC 4MB（实际 ~1700 字节）按 64 字节对齐**，不同线程的 TLC 不会在尾部共享 cache line。多线程下 T=8 64B churn 的 speedup 从 0.95-1.16× 拉到 1.50-2.55×。

`objdump` 验证 mp_malloc 现在是 `shl $0x6, %rax`：

```
2924:   48 c1 e0 06          shl    $0x6,%rax
```

## 六、为什么之前没发现

1. **docs/性能指标测试.md 是 Win UCRT 上跑的**。Win 路径的 `tlc_current()` 只调一次 `TlsGetValue`（src/tlc.cpp:30-33），没有 `pthread_once`，所以 mempool 自己代码就比 POSIX 版快一半。Win UCRT 这边 mempool fast path 跟 UCRT free list 量级接近，看不出 TLS 实现问题。
2. **glibc 2.39 tcache 把 `tcache_get` / `tcache_put` 标了 `__always_inline`**，每次 malloc/free 走单条 `mov %fs:offset` + 几条直线指令。这是非常硬的 baseline。POSIX 路径的 mempool 多两次 PLT 跳转就被照出来了。
3. **`tests/` 目录下原有 microbench 是 batch 模式**（一次 alloc N 个再一次 free N 个），不会触发 churn 路径反复进 `tlc_alloc_slow`。Table 2 churn 模式是 v1.21.0 的 `real_tests/` 才补上的。

## 七、代价与适用范围

- **`tls_model("initial-exec")` 限制**：libmempool.so 必须随主程序启动加载（普通 `-lmempool` 链接就 OK），**不能 dlopen 后再加载，也不能用作 LD_PRELOAD malloc 替换**。这两种用法时需要去掉 `tls_model` attribute，回退到 General Dynamic（每次访问 `__tls_get_addr`，约 20ns 额外开销/次，churn 速度回到 v1.20.0 量级）。
- **`alignas(64)` TLC** 让每个 TLC 多 ~50 字节内存（1700 → 1728 + padding），单线程 100 个 TLC 多 5KB，无关紧要。
- **fast path 头内联**让公共头 `tlc.h` 暴露 `Bucket`/`TLC` 全部字段给 caller TU，扩了内部 ABI 暴露面。`Bucket`/`TLC` 已经在 `tlc.h` 中定义，没有 ABI 新泄露，但下次改 struct layout 要更小心（外部翻译单元可能 inline 了它）。
- **fast path 不再走 `bucket_collect_local`**，意味着每 64 次 collect 触发一次的 amortized reclaim 不再发生在 churn 路径。Reclaim 现在只在 `tlc_alloc_slow` 拿新页时触发（每 64 次新页分配跑一次）。Churn 工作集驻留时本来就不需要 reclaim，无影响；高分配-高释放循环（如 game scenario）仍会周期性触发 reclaim。

## 八、和 v1.20.0 早先 docs 的关系

之前的 `docs/v1.20.0_实测重跑.md`（已删）把 K=256 churn 输 glibc 解释为"glibc 2.39 tcache 是世界级最强基线，mempool 在 Win UCRT 上赢是因为 UCRT 弱"。**这个解释在没有 perf 数据时是猜测，被本次 perf record 证伪**：真正原因是 mempool POSIX 路径每次 alloc/free 都跑两次 libc PLT，glibc 跑零次。修了 TLS 之后 mempool 在 K=256 churn 上反超 glibc 1.62-3.62×。

教训：性能问题不要在没有 profiler 数据时编原因。`perf record + perf annotate` 在 VMware 没有硬件 PMU 的环境里仍然能给出可靠的 hot function 分布（用 `cpu-clock` 软件事件）。

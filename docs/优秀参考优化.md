# 优秀分配器参考优化记录

本文档记录 mempool 参考业界优秀内存分配器进行的每次迭代优化，包含参考来源、设计思想、代码改动和性能数据。

---

## 迭代 1: v1.2.0 — Temporal Cadence（来自 mimalloc）

### 参考来源
- **项目**: [microsoft/mimalloc](https://github.com/microsoft/mimalloc)
- **关键文件**: `src/alloc.c` — `mi_free_block_local()`
- **核心思想**: mimalloc 将 free 路径做到极简（仅 push 到 local free list），合并操作延迟到 alloc 时 free_head 耗尽才触发。这种设计称为 "temporal cadence"——free 和 alloc 在时间上自然交替，合并开销被 alloc 的慢路径自然摊销。

### 问题分析
原来的 `tlc_free` 每次 free 都执行 watermark 检查：
```cpp
if (MP_UNLIKELY(b.local_free_count >= MP_HIGH_WATERMARK)) {
    bucket_collect_local(&b);
}
```
虽然用了 `MP_UNLIKELY` 提示分支预测，但在高频 free 场景下仍有额外的比较+分支指令开销。

### 代码改动
**文件**: `src/tlc.cpp` — `tlc_free()` 函数

**改动**: 删除 watermark 检查，free 路径变为纯粹的链表 push + 原子减 + 统计更新：
```cpp
void tlc_free(TLC* tlc, void* ptr, PageMeta* pm, uint32_t bucket_idx) {
    Bucket& b = tlc->buckets[bucket_idx];
    FreeBlock* blk = static_cast<FreeBlock*>(ptr);
    blk->next = b.local_free_head;
    b.local_free_head = blk;
    b.local_free_count++;
    pm->used_count.fetch_sub(1, std::memory_order_relaxed);
    tlc->stats.free_count++;
    tlc->stats.free_bytes += sc_block_size(bucket_idx);
    // 不再检查 watermark，合并由 alloc 慢路径触发
}
```

合并时机不变——`tlc_alloc` 在 `free_head` 为空时依次尝试 `bucket_collect_local` → `bucket_collect_thread_free` → `bucket_alloc_new_pages`。

## 迭代 2: v1.3.0 — Bump Pointer 初始化（来自 snmalloc + LevelDB）

### 参考来源
- **项目**: [microsoft/snmalloc](https://github.com/microsoft/snmalloc), [google/leveldb](https://github.com/google/leveldb)
- **关键文件**: snmalloc `src/mem/freelist.h` — bump-then-freelist 混合策略; LevelDB `util/arena.cc` — 线性 bump allocator
- **核心思想**: 新页面不预建完整 free list（O(N) 遍历所有 block 串链表），而是用 bump pointer 按需分配。只需一次指针加法即可返回下一个 block，无需任何链表操作。块被释放后进入 local_free_head，在 free_head 耗尽时被回收。

### 问题分析
原来 `init_multi_page_freelist()` 在分配新页面时遍历所有 block 反向串链表：
```cpp
for (int p = num_pages - 1; p >= 0; p--) {
    for (int i = blk_per_page - 1; i >= 0; i--) {
        blk->next = head; head = blk;
    }
}
```
对于大 block_size（如 4096B，每页仅 1 个 block），开销不大；但对于小 block_size（如 16B，每页 256 个 block），每次请求新页面都要做 256 次链表串接。

### 代码改动
**文件**: `src/tlc.h` — Bucket 结构增加 bump 字段
```cpp
struct Bucket {
    // ... existing fields ...
    uint8_t*  bump_ptr;    // next block to allocate via bump
    uint8_t*  bump_limit;  // end of current bump region
    // ...
};
```

**文件**: `src/tlc.cpp` — 三处改动：
1. 删除 `init_multi_page_freelist()`
2. `bucket_alloc_new_pages()` 设置 bump_ptr/bump_limit 替代建 free list
3. `tlc_alloc()` 增加 bump 快速路径（free_head miss → bump alloc）

### 性能数据对比

### 总结
Bump pointer 消除了首次分配时的 O(N) free list 初始化，alloc 路径仅需一次指针加法。大 block 场景提升最为显著（128B~4096B 首次 alloc 提速 66%~93%），多线程混合场景也受益明显。

## 迭代 3: v1.4.0 — 跨线程 free O(1) 直达（来自 mimalloc + snmalloc）

### 参考来源
- **项目**: [microsoft/mimalloc](https://github.com/microsoft/mimalloc), [microsoft/snmalloc](https://github.com/microsoft/snmalloc)
- **核心思想**: mimalloc 每个 page 直接关联其 owner heap，跨线程 free 时无需查找 owner，直接 CAS push 到目标 page 的 thread_free list。snmalloc 则通过消息队列实现类似效果。

### 问题分析
原来的跨线程 free 路径需要：lock(arena->tlc_mutex) → 遍历 TLC 链表 → 比较 thread_id → CAS push。O(N) 复杂度（N=线程数），且有 mutex 争用。

### 代码改动
1. `arena.h`: PageMeta 增加 `TLC* owner_tlc` 字段
2. `tlc.cpp`: `bucket_alloc_new_pages()` 分配后设置 `owner_tlc`
3. `mempool.cpp`: `mp_free()` 跨线程路径直接用 `pm->owner_tlc` 做 CAS push，无锁无遍历

### 性能数据

| 指标 | v1.3.0 | v1.4.0 | 提升 |
|------|--------|--------|------|
| 跨线程 free (frees/ms) | 86,147 | 106,975 | +24% |
| 交替 16B (ops/ms) | 46,788 | 67,342 | +44% |
| 8线程混合加速比 | x2.50 | x2.73 | +9% |

## 迭代 4: v1.5.0 — Size Class O(1) 查找表（来自 tcmalloc + rpmalloc）

### 参考来源
- **项目**: [google/tcmalloc](https://github.com/google/tcmalloc), [mjansson/rpmalloc](https://github.com/mjansson/rpmalloc)
- **核心思想**: tcmalloc 和 rpmalloc 都使用预计算查找表将 size → size_class 的映射做到 O(1) 无分支。

### 代码改动
**文件**: `src/size_class.h`
- 编译期生成 257 项 `uint8_t` 查找表（`constexpr`）
- `sc_index_of(size)` 从二分搜索（3-4 次循环）变为单次数组访问：`return sc_table.data[(size + 15) / 16]`

### 总结
查找表对中大尺寸（>128B）的 alloc+free 路径有稳定 4%~9% 提升，消除了二分搜索的分支预测开销。

## 迭代 5: v1.6.0 — used_count 原子操作移除（来自 mimalloc）

### 参考来源
- **项目**: [microsoft/mimalloc](https://github.com/microsoft/mimalloc)
- **核心思想**: mimalloc 的 page 级 used_count 只在页面回收时更新，不在每次 alloc/free 的热路径中做原子操作。分析发现 mempool 的 `used_count` **仅被写入，从未被读取做条件判断**，因此完全可以从热路径中移除。

### 代码改动
- `tlc_alloc()`: 移除两处 `chunk->pages[pi].used_count.fetch_add(1)` 原子操作
- `tlc_free()`: 移除 `pm->used_count.fetch_sub(1)` 原子操作
- `mp_free()` 跨线程路径: 移除 `pm->used_count.fetch_sub(1)` 原子操作

### 性能数据

#### 交替 alloc+free（最能体现此优化的场景）

| Size | v1.5.0 ops/ms | v1.6.0 ops/ms | 提升 |
|------|---------------|---------------|------|
| 16B  | 45,160        | 61,282        | +36% |
| 32B  | 65,049        | 75,053        | +15% |
| 64B  | 64,761        | 72,432        | +12% |
| 128B | 62,141        | 73,236        | +18% |
| 256B | 63,599        | 72,516        | +14% |
| 512B | 62,416        | 74,316        | +19% |

#### 批量 alloc+free

| Size | v1.5.0 加速比 | v1.6.0 加速比 | 提升 |
|------|---------------|---------------|------|
| 16B  | x1.37         | x1.66         | +21% |
| 64B  | x1.72         | x1.87         | +9%  |
| 128B | x2.24         | x2.35         | +5%  |
| 256B | x2.41         | x2.74         | +14% |

#### 多线程扩展

| 指标 | v1.5.0 | v1.6.0 | 提升 |
|------|--------|--------|------|
| 4线程 64B ops/ms | 67,002 | 93,917 | +40% |
| 8线程 64B ops/ms | 87,547 | 97,324 | +11% |

### 总结
移除 alloc/free 热路径的原子操作是最高性价比的优化。每次 alloc 节省 ~5ns（x86 `lock xadd`），每次 free 也节省 ~5ns。交替场景（alloc+free 各一次原子操作）提升最大（+15%~+36%）。

## 迭代 6: v1.7.0 — 热路径统计精简 + 死代码消除

### 参考来源
- 通用微优化原则：减少热路径指令数
- mimalloc: 热路径中最少的内存写入

### 代码改动
1. 移除 Bucket 中的 `free_count` 和 `local_free_count`（从不用于条件判断，纯死追踪）
2. 移除 `fast_path_hits++` 从热路径（改为 `alloc_count - slow_path_hits` 在 aggregate 时推算）
3. 移除 `tlc_free` 中的 `pm` 参数使用（unused after used_count removal）

### 热路径优化前后对比
```
alloc 快速路径（free_head pop）:
v1.6.0: free_count--, alloc_count++, alloc_bytes+=, fast_path_hits++  (4 writes)
v1.7.0: alloc_count++, alloc_bytes+=                                   (2 writes)

free 路径:
v1.6.0: local_free_count++, free_count++, free_bytes+=                (3 writes + 链表push)
v1.7.0: free_count++, free_bytes+=                                     (2 writes + 链表push)
```

|------|--------|--------|------|
| 交替 16B ops/ms | 61,282 | 79,246 | +29% |

## 迭代 7: v1.8.0 — mp_free 快速路径内联优化

### 核心改动
1. **消除 `get_thread_id()`**: 用 `pm->owner_tlc == tlc_current()` 代替 `pm->owner_thread == get_thread_id()`，避免系统调用
2. **减少 `tlc_current()` 调用**: 同线程和跨线程路径共用一次调用结果
3. **内联 `resolve_block_ptr()`**: 直接在 `mp_free` 中内联 chunk_of + page_index 计算，避免函数调用开销

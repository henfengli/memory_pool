# v1.20.0 Reclaim 不变量修复 — 方案 D（reclaim 时按页统计）

日期: 2026-05-01
触发: 用户与 LLM 对话时审视一份 AI 报告里说的"freed_count 在 alloc 弹出时未递减"的 bug。核查源码确认 bug 真实存在；首版修复（"方案 A，alloc fast path 1 加 freed_count--"）实测带来 14~22%（部分场景 30%）的吞吐损失，被用户拒绝。改用**方案 D：移除 PageMeta::freed_count 字段，reclaim 时直接遍历 free 链表按页统计**。

## Bug 现象

长期 churn 同一 size class（持有大量 live block + 反复 alloc/free 一个热点 block）时，[bucket_try_reclaim_pages](../src/tlc.cpp) 的判定条件被破坏，把仍持有 live block 的页面归还 arena，arena 把该页重新分配给其他 size class，导致 live block 内存被覆盖。

### 实测复现（test_reclaim_invariant，96B size class，每页 42 block）

```
v1.19.0 (BUG):
  [CORRUPT] live[7980] first byte=00 expected=2c
  [CORRUPT] live[7981] first byte=00 expected=2d
  ...
  [FAIL] 20 / 8000 live blocks were corrupted
  exit_code=3

v1.20.0 (FIX):
  [STEP 1] allocated 8000 live blocks of 96 bytes each
  [STEP 2] churned 200000 alloc-free cycles
  [STEP 3] all 8000 live blocks verified intact
  [STEP 4] freed all live blocks without crash
  [PASS] reclaim invariant holds under sustained churn
```

## 根因（v1.19.0 设计缺陷）

v1.19.0 的 `PageMeta::freed_count` 是**只增不减的累计计数**：
- 同线程 free → `++`
- 跨线程 free 由 collect_thread_free 阶段调用 `compensate_freed_counts` 补偿 `++`
- alloc 弹出 → **不动**

reclaim 判定 `freed_count == blocks_per_page` 设计意图是"页面全空"。但在 churn 模式下每次 free `++` 不被 alloc 抵消，`freed_count` 单调超过 `blocks_per_page`，触发 reclaim 误判。

## 修复历程

### 首版尝试（已废弃）：方案 A，alloc fast path 加 freed_count--

```cpp
// fast path 1 加入：
ChunkHeader* _ch = chunk_of(blk);
_ch->pages[page_index_of(_ch, blk)].freed_count--;
```

把 `freed_count` 语义改为「当前在 free / local_free / thread_free 任一链表上的 block 数」。修复正确但**性能损失 14~22%**（test_perf_detailed 实测）。

理由：本项目 fast path 1 的中心是 `Bucket`（一条 cache line 内完成所有读写），PageMeta 在另一条 cache line 上。`freed_count--` 引入了**额外的 cache line load + store**，且每次 alloc 都要算 `chunk_of` + `page_index_of`。

参照 git 历史 `e521bab` (v1.6.0)："移除 used_count 原子操作（参考 mimalloc）—— 仅被写入从未被读取判断，移除 alloc/free 热路径的 fetch_add/fetch_sub。**交替场景提升 15~36%，4 线程扩展性提升 40%**"——v1.6.0 当年正是因为同样的开销才把 used_count 删掉。方案 A 等于退回到 v1.6.0 之前的状态。

### 工业参照：mimalloc 为什么 used++ 不慢？

[mimalloc/src/alloc.c](https://github.com/microsoft/mimalloc/blob/master/src/alloc.c) 的 fast path 确实有 `page->used++`。但 mimalloc 的 fast path 中心数据结构就是 `mi_page_t`：`page->free` 读、`page->free = next` 写、`page->used++` 都在同一条 cache line。**对 mimalloc 是 0 cache line 增量**；本项目的 hot path 中心是 `Bucket`，PageMeta 是冷数据，**强行往里塞写就是新增 cache line 访问**。

[mimalloc 设计文档](https://github.com/microsoft/mimalloc/blob/master/include/mimalloc/types.h) 直接写明：
> `used - |thread_free| == actual blocks that are in use (alive)`
> We **don't count `freed`** but use `used` to **reduce the number of memory accesses** in `mi_page_all_free`.

mimalloc 选 `used` 不选 `freed_count`，理由就是「减少内存访问」。本项目 hot path 架构与 mimalloc 不同，对应的最优解是**完全不在 hot path 维护 page 级计数**。

### 最终方案 D：reclaim 时按页统计

**核心思想**：取消 `freed_count` 字段，reclaim 触发时直接遍历 `free_head` + `thread_free`（atomic exchange 摘走）链表，按页统计 block 数。某 PageRange 上每页 block 数 == `blocks_per_page` ⇒ 全空 ⇒ 回收。

#### 字段层面改动

| 项 | v1.19.0 | v1.20.0 |
|---|---|---|
| `PageMeta::freed_count` | 存在（uint16_t）| **删除** |
| `PageMeta` 总尺寸 | 12B（MinGW32）| 12B（用 4B `_pad` 占位保持对齐）|

PageMeta 实际大小不变（编译器对齐），但语义上少了一个易错字段。

#### 路径层面改动

| 路径 | v1.19.0 | v1.20.0 |
|---|---|---|
| **alloc fast path 1**（free_head pop）| 不动 freed_count | 不动（与 v1.19.0 一致，零额外开销）|
| **alloc fast path 2**（bump 切出）| 不动 | 不动（与 v1.19.0 一致）|
| **tlc_free**（同线程释放）| `pm->freed_count++` | **删除**（不再写 PageMeta）|
| **tlc_free_remote**（跨线程释放）| 仅 CAS push | 仅 CAS push（与 v1.19.0 一致）|
| **bucket_collect_thread_free** | exchange + compensate_freed_counts | exchange，splice 到 free_head（删除 compensate）|
| **bucket_try_reclaim_pages** | 读 PageMeta::freed_count 判定 | **遍历 free_head 数 block 判定**（核心改动）|
| **tlc_flush** | exchange + compensate_freed_counts | exchange，丢弃（页面整体归还，链表节点失效）|
| **compensate_freed_counts 函数** | 存在 | **删除**（无用）|

#### 新版 reclaim 算法（[src/tlc.cpp](../src/tlc.cpp) `bucket_try_reclaim_pages`）

```cpp
static void bucket_try_reclaim_pages(Bucket* b, Arena* arena) {
    // 1. 摘走 thread_free 拼到 free_head（顺便完成本轮 collect_thread_free）
    FreeBlock* tf = b->thread_free_head.exchange(nullptr, std::memory_order_acquire);
    if (tf) { /* splice into free_head */ }

    // 2. 对每个 PageRange：数 free_head 上属于该 range 的 block 数
    //    若 == blocks_per_page * pr->count 则该 range 全空，回收。
    for each PageRange pr:
        if (bump_ptr 还指着 pr 内) skip;
        uint32_t blocks_seen = count_blocks_in_range(b->free_head, pr->ptr, pr->count);
        uint32_t expected = sc_info(b->bucket_idx).blocks_per_page * pr->count;
        if (blocks_seen < expected) skip;
        过滤 free_head 移除 pr 内的 block;
        arena_free_pages(arena, pr->ptr, pr->count);
        移除 pr;
}
```

#### 并发安全（关键论证）

reclaim 在 `atomic exchange(thread_free_head, nullptr)` **之后**才统计。考虑两种竞争：

1. **跨线程 free 在 exchange 之前 push**：block 已在被摘走的链表里，被统计 → 正确。
2. **跨线程 free 在 exchange 之后 push**：意味着该用户线程在那个时刻**仍持有该 block**——也就是说 page 上至少有 1 个 live block 没被 free → 该 page 的 `blocks_seen < expected` → reclaim 跳过该 page → **正确**。

late-arriving 的 push 留在新的 thread_free_head 上，下次 collect_thread_free 处理。

## 路径开销对比表

| 路径 | v1.19.0 | v1.20.0 旧（方案 A）| v1.20.0 新（方案 D）|
|---|---|---|---|
| tlc_alloc fast path 1 | 3 条内存操作（Bucket）| +chunk_of+page_index_of+PageMeta 写 | 3 条内存操作（与 v1.19.0 一致）|
| tlc_alloc fast path 2（bump）| Bucket 读写 | Bucket 读写 | Bucket 读写 |
| tlc_free 同线程 | Bucket 写 + PageMeta 写 | Bucket 写 + PageMeta 写 | **仅 Bucket 写**（PageMeta 加载已由 mp_free 完成，但不再写）|
| tlc_free_remote 跨线程 | 1 次 CAS | 1 次 CAS | 1 次 CAS |
| collect_thread_free | exchange + 遍历 ++ | exchange + 遍历 ++ | **仅 exchange + splice**（无需遍历）|
| reclaim | 读 N 个 PageMeta | 读 N 个 PageMeta | 遍历 free_head 一次（每 PageRange 一次，已存在的过滤循环复用）|
| reclaim 触发频率 | 每 64 次 collect | 每 64 次 collect | 每 64 次 collect |

**收益**：方案 D 不仅修了 bug，还**删掉了一次 free 路径上的 PageMeta 写**（tlc_free）和**一次 collect 路径的 list 遍历**（compensate_freed_counts）。

## 顺手砍掉的冷开销

审视 hot path 时确认并完成的清理：

1. **删除 `compensate_freed_counts` 整个函数**：13 行，函数体含一次链表遍历 + 每节点 chunk_of/page_index_of/PageMeta 写。曾在 collect_thread_free 和 tlc_flush 各调用一次，现两处都不再需要。
2. **`tlc_free` 删除 `pm->freed_count++`**：每次同线程 free 少一次 PageMeta 写。`pm` 参数保留以保持 ABI 稳定（mempool.cpp 已有 pm，零成本传递）。
3. **`tlc_flush` 不再补偿计数**：直接 exchange 丢弃 thread_free（页面整体归还，节点失效）。

## 借助这个 bug 学到的工程教训

1. **设计层的形式化论证若未经压力测试，前提假设可能被破坏**。v1.19.0 的 freed_count 设计在"短时 churn 不触发 reclaim"的隐含假设下成立，但作者从未在 long-running churn 场景下压测。
2. **fast path 的 cache line 性质决定可塞什么**。同样是"alloc 时维护 page 级计数"，mimalloc 几乎免费（fast path 中心就是 page），本项目损失 30%（fast path 中心是 Bucket，PageMeta 是冷数据）。**直接照搬其他分配器的设计不一定 work**。
3. **冷路径解决方案优于 hot path 解决方案**。reclaim 已经在每 64 次 collect 才触发的冷路径，把"一次性遍历计数"放进去几乎不增加渐进复杂度，比"hot path 维护增量"的总成本低数十倍。
4. **查 git 历史比读代码快**。如果一开始就看到 v1.6.0 commit message"移除 used_count 原子操作 ... 提升 15~36%"，根本不会走方案 A 的弯路。
